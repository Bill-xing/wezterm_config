# Ubuntu 22.04 终端环境落地设计

- 日期：2026-04-13
- 仓库：`/home/xing/project/wezterm_config`
- 目标环境：`Ubuntu 22.04`
- 主题：按当前仓库既有方案完整落地终端工作流配置

## 背景

当前仓库维护了一套跨平台 dotfiles，覆盖 `zsh`、`tmux`、`Neovim`、`Yazi`、`WezTerm` 等终端工作流组件，并通过 `install/ubuntu.sh` 与 `bootstrap/common.sh` 定义 Ubuntu 的标准安装路径、来源、固定版本插件和配置文件落位方式。

本次工作不重新设计仓库结构，也不改写已有配置策略，而是将当前仓库定义的 Ubuntu 方案完整落地到一台 `Ubuntu 22.04` 机器，并完成基础可用性验证。

## 目标

1. 按仓库现有逻辑在 `Ubuntu 22.04` 上安装终端工作流所需组件。
2. 保持安装来源、固定版本插件、配置目录结构与仓库现状一致。
3. 全流程显式使用代理 `127.0.0.1:10808` 处理联网步骤。
4. 在安装完成后执行基础验证，确认核心工具已可用。

## 非目标

- 不修改仓库中现有配置的功能或键位设计。
- 不更换插件版本、下载来源或安装目标路径。
- 不永久写入系统级代理配置，除非后续另行提出。
- 不把首次启动时的在线插件拉取纳入本轮“必须完成”的同步安装范围。

## 当前仓库约束

本次落地需要遵循以下仓库现状：

- Ubuntu 入口脚本为 `install/ubuntu.sh`
- 通用安装逻辑位于 `bootstrap/common.sh`
- 锁定版本的 zsh/tmux 相关第三方组件位于 `bootstrap/plugins.lock.sh`
- 配置文件落位方式由 `install_config_payload` 决定，默认在 Unix 上使用软链接
- 目标配置文件包括：
  - `~/.zshrc`
  - `~/.p10k.zsh`
  - `~/.tmux.conf`
  - `~/.wezterm.lua`
  - `~/.config/nvim`
  - `~/.config/yazi`

## 推荐方案

采用“分阶段执行、统一验证”的方式落地当前仓库配置。

该方案仍以仓库现有安装脚本和目录结构为准，但不会直接一把执行完整脚本后再被动排错，而是将过程拆成可观察的阶段，确保在代理、权限、第三方源、GitHub Release 下载、配置链接等关键节点都能明确判断状态并快速定位问题。

相比直接运行脚本，此方式更适合当前任务，因为本次目标包含：

- 需要显式经由代理联网
- 需要在 `Ubuntu 22.04` 上完成基础验证
- 需要在失败时明确区分网络、权限、上游资源、配置文件等不同问题来源

## 执行架构

整体执行遵循“仓库定义不变，执行方式分阶段”的原则：

1. 以 `install/ubuntu.sh` 和 `bootstrap/common.sh` 为主线确认实际安装范围。
2. 在命令执行环境中注入代理，而不是修改仓库逻辑或永久改写用户 shell 配置。
3. 按阶段安装系统包、用户态工具、第三方仓库与固定版本插件。
4. 最后调用仓库既有的配置落位逻辑，让所有配置统一由仓库管理。
5. 以命令级验证作为交付标准，而不是仅以“脚本跑完”作为成功判定。

## 分阶段执行顺序

### 阶段 1：环境与前提检查

确认以下条件成立：

- 当前系统为 `Ubuntu 22.04`
- 拥有 `sudo` 权限
- 具备仓库脚本要求的基础命令，例如 `git`、`curl`、`python3`
- 本地代理 `127.0.0.1:10808` 可达

这一阶段只做确认，不修改仓库内容。

### 阶段 2：系统基础包安装

按 `install/ubuntu.sh` 的既有列表安装基础包，包括但不限于：

- `build-essential`
- `ca-certificates`
- `curl`
- `file`
- `fzf`
- `fd-find`
- `git`
- `gnupg`
- `jq`
- `ripgrep`
- `tmux`
- `unzip`
- `wget`
- `zsh`

若当前 apt 源中存在 `bat`、`eza`、`direnv`，则按脚本逻辑一并安装。

### 阶段 3：用户态工具安装

按仓库已有函数安装：

- `Neovim` 最新官方 release 预编译包到 `~/.local/opt/nvim`
- `Lazygit` 官方 GitHub Release
- `Yazi` 官方 GitHub Release

同时按仓库逻辑补齐 Ubuntu 兼容软链：

- 如果系统只有 `batcat`，则创建 `~/.local/bin/bat`
- 如果系统只有 `fdfind`，则创建 `~/.local/bin/fd`

### 阶段 4：WezTerm 官方仓库安装

按仓库既有流程：

1. 配置 WezTerm 官方 apt 仓库
2. 更新包索引
3. 安装 `wezterm`

此阶段单独观察其仓库配置与安装结果，因为它同时涉及第三方源、系统级安装和网络访问。

### 阶段 5：固定版本插件与配置落地

按 `bootstrap/plugins.lock.sh` 和 `bootstrap/common.sh` 完成以下内容：

- 安装固定版本的 `oh-my-zsh`
- 安装固定版本的 `powerlevel10k`
- 安装固定版本的 `zsh-autosuggestions`
- 安装固定版本的 `zsh-syntax-highlighting`
- 安装固定版本的 tmux 插件：
  - `tpm`
  - `tmux-sensible`
  - `tmux-yank`
  - `tmux-resurrect`
  - `tmux-continuum`
  - `tmux-powerline`

随后通过仓库已有 `install_config_payload` 逻辑，把配置文件软链接到用户目录与 `~/.config`。

### 阶段 6：基础验证

安装完成后执行以下验证：

- `zsh -i -c exit`
- `tmux source-file ~/.tmux.conf`
- `nvim --version`
- `yazi --version`
- `lazygit --version`
- `wezterm -V`

如存在 `bat`/`fd` 兼容软链需求，也需确认对应命令可执行。

## 代理策略

本次落地要求全流程显式使用代理 `127.0.0.1:10808`。

执行策略如下：

- 在命令执行时临时注入：
  - `http_proxy=http://127.0.0.1:10808`
  - `https_proxy=http://127.0.0.1:10808`
  - `all_proxy=socks5://127.0.0.1:10808`
- 需要联网的命令统一继承这些环境变量，例如：
  - `apt-get`
  - `curl`
  - `wget`
  - `git clone`
  - `git fetch`
- 默认不将代理永久写入 `~/.zshrc`、`/etc/apt/apt.conf.d/` 或其他系统配置位置

该策略既满足本次安装需求，也避免为后续日常环境引入隐式副作用。

## 备份与幂等策略

沿用仓库已有逻辑，不另发明备份机制：

- 若目标配置文件或目录已存在，使用 `backup_existing` 生成带时间戳的备份
- 若目标已经是指向仓库文件的正确软链，则跳过重复处理
- 配置落位阶段允许安全重试

这保证了本次落地不会无提示覆盖已有个人配置。

## 失败处理策略

若执行过程中出现问题，按阶段定位并停止在最小故障面：

- 基础包阶段失败：优先判断 `sudo`、apt 源和代理是否正常
- GitHub Release 下载失败：优先判断代理继承、下载地址访问和网络超时
- 第三方源阶段失败：优先判断 WezTerm 官方仓库配置是否成功写入以及 apt 更新是否可达
- 插件克隆失败：优先判断 git 代理和 GitHub 连通性
- 配置落位失败：优先判断目标路径权限和现有文件备份结果
- 验证阶段失败：仅将失败组件单独标记，不把全部工作误报为完全成功

本方案默认不执行“大范围回滚”。如出现失败，保留现场以便继续修复和重试。

## 验收标准

当满足以下条件时，本次工作判定为完成：

1. 核心组件已按仓库既有方案安装：
   - `zsh`
   - `tmux`
   - `nvim`
   - `yazi`
   - `lazygit`
   - `wezterm`
2. 关键配置文件已落位到用户目录和 `~/.config`
3. 基础验证命令均成功返回或输出合理版本信息
4. 若存在首次启动才触发的在线步骤，例如 `LazyVim` 拉取插件，需在交付说明中明确提示

## 交付结果

交付时需要向用户汇报：

- 各阶段是否成功完成
- 哪些配置文件已软链接或复制到目标位置
- 哪些核心命令已经验证通过
- 是否存在剩余注意事项，例如默认 shell 切换、首次启动联网拉插件、字体未安装等

## 后续实施边界

本设计文档只定义“如何按当前仓库在 `Ubuntu 22.04` 上完成配置落地”，不直接实施安装。  
实现阶段应在不偏离仓库现有安装逻辑的前提下，编写一个具体执行计划，并按计划逐步执行与验证。
